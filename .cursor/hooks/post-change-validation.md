# Post-Change Validation Hook

## Мета

Після змін швидко перевірити, що не внесено очевидних регресій.

## Кроки

1. **Збірка / типи**: якщо в проєкті є `pnpm lint`, `npm run build` або `tsc --noEmit` — переконатися, що зміни не ламають типи та лінтер (або зафіксувати, що команда ще не підключена).
2. **`any`**: переконатися, що в змінених файлах не з’явилися нові `any` без обґрунтування.
3. **Service layer**: для змін у домені клієнтів/записів/авторизації — чи залишився виклик даних за межами UI (через сервіси)?
4. **API Routes / Route Handlers** (якщо є): перевірити коди відповіді, форму успіху/помилки, обробку винятків; не витікати внутрішніми деталями в production.
5. **`middleware.ts` / редіректи**: перевірити сценарії login/logout, захищені `/dashboard/*`, публічні сторінки.
6. **Auth / billing** (коли з’явиться): ключові потоки — логін / сесію / вихід; оплата — checkout / success / cancel.
7. **Supabase RLS**: якщо змінювались запити або схема — чи не відкрили доступ до чужих рядків?

## Вихід

- Що перевірено (пунктами).
- Що залишається ризиком або вимагає ручного тесту в браузері.
