# Pre-Change Hook

## Мета

Перед тим як змінювати код, звузити ризик зайвих правок і порушення архітектури Plurio (service layer, ізоляція даних).

## Кроки

1. **Скоуп**: що саме змінюється (UI, сервіс, міграція, публічна сторінка)? Чи потрібні зміни в кількох шарах?
2. **Джерело даних**: чи залишається доступ до БД лише через `services/*` (або майбутній API-клієнт), без прямого Supabase у компонентах?
3. **Схема / RLS**: якщо чіпаєш таблиці або політики — чи заплановані міграція та перевірка `user_id` / tenant isolation?
4. **Публічні маршрути**: для `/u/[username]` та інших публічних URL — що саме видно без авторизації?
5. **Залежності**: чи можна перевикористати існуючі типи, хелпери та патерни з репозиторію замість нових абстракцій?

## Вихід

Коротко: що буде змінено, які файли зачіпаються, чи потрібні міграції або оновлення RLS.
