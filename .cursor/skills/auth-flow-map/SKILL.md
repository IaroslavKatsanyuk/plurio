# Карта потоків авторизації

## Шари

1. **Клієнт Supabase** — серверний і браузерний зі спільних хелперів `lib/supabase` (див. файли проєкту).
2. **Middleware** — за потреби оновлення / редірект для `/dashboard/*` проти публічних маршрутів.
3. **Сервіси** — `auth.service.ts` обгортає операції auth; UI викликає сервіси, а не сирий `supabase.auth` у компонентах, коли це можливо.
4. **RLS** — дані обмежені `auth.uid()` / `user_id`; не обходь RLS з клієнта через service role.

## Потоки, які варто зберегти

- Реєстрація → підтвердження email (якщо увімкнено) → вхід → дашборд
- Вхід → cookie сесії → дашборд
- Вихід → очищення сесії → редірект на `/login`
- Прострочена сесія → оновлення або редірект на логін без нескінченних циклів

## Чеклист

- [ ] Секрети auth не в `NEXT_PUBLIC_*`
- [ ] Callback / OAuth URL узгоджені з дашбордом Supabase і env
- [ ] Захищені сторінки не «мигають» неправильним контентом до редіректу (за потреби UI завантаження)
